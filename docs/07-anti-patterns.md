## 第七部分：反模式与防御性工程

### 7.1 指令极性反模式

**核心反模式**：在推理模型上使用负向约束作为主要的行为控制手段。

实证数据支撑：纯负向约束的违背率约 12%，正向替代方案仅约 3%。更严重的是，在安全对齐过程中，大规模的负向约束会触发过度拒绝（Over-refusal）危机——OR-Bench 80,000 测试样本的评测表明，现有对齐技术在提升抗恶意能力的同时付出了惨痛的误拒代价。

在推理模型上，这一反模式的代价更高。GPT-5.4 会将推理 Token 浪费在试图理解和调和矛盾的负向指令上；Gemini 3.1 Pro 可能"过度聚焦"于负向约束而无法执行基本逻辑。

**正确做法**：将所有负向约束转化为正向建设性要求；对不可避免的安全底线，使用物理隔离（分隔符包裹）和"前后书挡化"策略。

### 7.2 推理模型特有反模式

**反模式一：对推理模型显式诱导 CoT**

在推理模型上添加"让我们一步步思考"或强制展示推理步骤，会导致 API 响应时间激增 20%-80% 而无准确率提升。推理模型在内部已经自动执行了基于强化学习的思维搜索，外部 CoT 诱导迫使模型将内部高维推理翻译成低效的人类自然语言。

**反模式二：过度工程化提示词**

三大平台同时警告的反模式。推理模型的"手术刀精度"意味着它们会认真对待提示词中的每一个词——冗余的框架、不必要的角色描述、过多的约束条件都会消耗推理资源。

- Anthropic：Claude 4.6 有"过度工程化倾向——创建额外文件、添加不必要的抽象、构建未被请求的灵活性" 。
- OpenAI：GPT-5.4 的"手术刀精度"意味着矛盾指令比传统模型造成更大伤害 。
- Google：Gemini 3.1 Pro 会"过度分析冗长的传统提示词技术" 。

**反模式三：无约束角色扮演**

在推理模型上使用无结构化约束的角色设定（如单纯的"你是一名专家"），不仅无法提升逻辑正确性（13.78% 的问题因角色引入而退化），还会消耗推理模型的内部推理 Token。正确做法是用约束优先的参数化规约替代角色头衔。

**反模式四：技巧堆叠**

同时使用角色设定 + CoT + few-shot + 条件约束等多种技巧的叠加，在传统模型上已被证实可能引发注意力冲突，在推理模型上这一问题更加严重——每种技巧都会被推理模型认真分析和处理，导致推理 Token 在多个维度上分散。

### 7.3 平台迁移反模式

**从传统模型迁移到推理模型时的常见陷阱**：

| 迁移场景 | 反模式 | 正确做法 |
| --- | --- | --- |
| 从 GPT-4.1 → GPT-5.4 | 保留所有详细的步骤指令 | 从 reasoning_effort=none 开始，按需提升 |
| 从 Claude 3.x → Claude 4.6 | 保留 prefill 响应 | prefill 已弃用；用直接指令替代 |
| 从 Claude 3.x → Claude 4.6 | 保留"CRITICAL: You MUST use this tool"等激进语言 | 回调至正常提示，如"Use this tool when..." |
| 从 Gemini 2.5 → Gemini 3.1 Pro | 保留 temperature < 1.0 | 移除温度设置，使用默认 1.0 |
| 从 Gemini 2.5 → Gemini 3.1 Pro | 保留复杂的 CoT 提示词工程 | 使用 thinking_level="HIGH" + 简化提示 |
| 从 o3 → GPT-5.4 | 直接替换 | 使用 medium 或 high reasoning_effort |
