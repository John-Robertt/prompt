## 第三部分：统一提示词生成架构（UPGA）

在建立了底层认知机制的理解（第一部分）和跨平台共识的提取（第二部分）之后，本部分将两者转化为一个可操作的统一提示词构建方法论——**统一提示词生成架构**（Unified Prompt Generation Architecture, UPGA）。UPGA 是一个由五个规范层组成的组合式框架，每一层都有明确的功能定义、跨平台映射和底层机制支撑。

### 3.1 理论基础：提示词作为组合结构

UPGA 的设计理念借鉴了语言学中的组合性原理（Principle of Compositionality）：一个良构的提示词由有限数量的功能模块组成，每个模块服务于模型处理管线中的特定功能，模块之间的组合关系决定了提示词的整体效力。

这一思路与第 1.5 节揭示的归纳头组合结构机制高度契合。模型通过组合多个自注意力层来提取抽象规律——如果提示词本身也是模块化组合的，那么模型的组合式处理管线就能够更高效地解析和利用提示词中的信息。UPGA 正是基于这一洞察，将提示词设计从"写一段自然语言文本"提升为"组装一个功能模块化的结构体"。

### 3.2 UPGA 的五个规范层

#### 层 1：意图声明层（Intent Declaration Layer）

**功能**：明确定义任务的终极目标、成功标准和范围边界。这一层回答的是"模型最终需要达成什么结果"的问题。

**跨平台映射**：
- **Anthropic**："清晰、明确地说明你期望的输出……把 Claude 想象成一位才华横溢但缺乏背景的新员工——你越精确地解释你想要什么，结果就越好" 。
- **OpenAI**："给出任务、约束和期望的输出格式……在指令中给出非常具体的成功响应参数，并鼓励模型持续推理和迭代直到匹配你的成功标准" 。GPT-5.4 特别强调"定义明确的输出契约"（output contract）和"明确的完成标准"（completion criteria） 。
- **Google**："清晰简洁地陈述你的目标……避免不必要的或过于劝说性的语言" 。

**构建指引**：

意图声明层应包含三个核心要素：

1. **任务目标**：用一到两句话描述最终需要达成的具体产出。使用动作动词（"重构"、"生成"、"分析"、"审查"），而非模糊的描述性语言（"帮我改进"、"看看这段代码"）。
2. **成功标准**：定义什么样的输出算作"完成"。在推理模型上，GPT-5.4 建议"尝试给出非常具体的成功响应参数，鼓励模型持续推理直到匹配你的成功标准" 。
3. **范围边界**：明确说明什么在范围内、什么不在。这对推理模型尤为重要——推理模型可能会过度探索（Claude Opus 4.6 被报告为"比前代模型做了显著更多的前期探索"），因此明确的范围边界能防止不必要的推理发散。

**底层支撑——逆向归纳法**：

意图声明层的设计原理直接来源于动态博弈论中的逆向归纳法（Backward Induction）。该方法从过程的最终阶段（即目标终端）开始倒推，递归地确定每个前置节点的最优决策。将其应用于提示词设计意味着：不是"想到什么写什么"，而是首先明确终极目标，然后从目标倒推出核心原则和约束条件。这一机制在第四部分（哲学基础）中将得到完整展开。

#### 层 2：上下文架构层（Context Architecture Layer）

**功能**：为模型提供完成任务所需的全部外部信息（参考文档、领域知识、环境约束），并以最优的物理布局组织这些信息以最大化模型的注意力效率。

**跨平台映射**：
- **Anthropic**："将长文档和输入放置在提示词的最上方"；使用 `<documents>` 父标签嵌套多个 `<document>` 子标签，附带 `<source>` 元数据和 `<document_content>` 实际内容 。
- **OpenAI**："提供参考文本"策略；GPT-5.4 的 1M Token 上下文窗口使整个代码库级别的分析成为可能 。
- **Google**："先提供所有上下文"；"在大段数据之后使用清晰的过渡短语来锚定上下文与查询之间的关系" 。

**构建指引**：

1. **静态内容前置**：将参考文档、代码库、API 文档等静态信息放置在提示词的最前部。
2. **结构化标签包裹**：使用 XML 或 Markdown 对不同类型的上下文进行物理隔离。
3. **动态查询后置**：将具体的任务指令和用户查询放置在提示词的最末尾。
4. **过渡锚定**：在大段上下文与查询之间使用"基于上述信息……"等锚定短语。

**底层支撑**：本层的设计直接基于第 1.3 节揭示的 U 型注意力衰减机制和"中间迷失"现象。静态内容前置确保其享有头部位置的高注意力权重；动态查询后置确保其享有尾部位置的位置编码优势；两者的结合形成了最优的"三明治"结构。

#### 层 3：行为规约层（Behavioral Specification Layer）

**功能**：定义模型输出的格式要求、风格参数和正向行为指令。这一层回答的是"输出应该是什么样的"的问题。

**跨平台映射**：
- **Anthropic**：提供了丰富的格式控制手段——"用正向格式描述替代否定指令"、XML 格式指示器、匹配提示词风格与期望输出风格 。
- **OpenAI**：通过 `verbosity` API 参数（low / medium / high）控制输出详细程度；推荐"定义明确的输出契约"（output contract） 。
- **Google**：建议"指定对阅读或生成响应的约束——你可以告诉模型应该做什么和不应该做什么，并要求特定的格式" 。

**构建指引**：

1. **正向约束优先**：所有行为要求首先以正向形式表达。根据实证数据，正向指令的违背率（约 3%）远低于负向指令（约 12%）。
2. **负向约束物理隔离**：当安全约束不可避免时，将其放置在由分隔符（如 `<constraints>` 或 `---`）包围的专门区块中，并在提示词首尾重复（"前后书挡化"策略）以对抗 U 型注意力衰减。
3. **约束优先 > 角色扮演**：用参数化的能力边界和格式约束替代虚拟的角色头衔。
4. **结构化框架辅助**：在复杂任务中，可借助 CO-STAR（Context / Objective / Style / Tone / Audience / Response）等框架确保所有必要维度不被遗漏。

**底层支撑**：本层整合了第 1.1 节（正向传播路径优势）、第 1.2 节（NTAS 否定语义流失）和第 1.4 节（角色设定注意力稀释）的全部底层机制。其中，将负向约束转化为正向说明后违背率从 12% 降至 3% 的实证数据，以及将非结构化自然语言替换为 JSON Schema 后迭代率从 38.5% 暴跌至 12.3% 的数据，为本层的构建指引提供了量化支撑。

#### 层 4：推理委托层（Reasoning Delegation Layer）——推理模型的核心差异化层

**功能**：配置模型的推理深度、自检机制和思考过程引导。**这一层是推理模型时代与传统模型时代的根本分界线**——在传统模型上，这一层的功能是"通过提示词文本诱导模型进行推理"；在推理模型上，这一层的功能转变为"通过 API 参数控制模型内置推理的深度，并通过提示词定义推理的方向"。

**推理模型时代的关键转变**：

传统策略（推理模型上的反模式）：
```
让我们一步一步地思考这个问题。首先分析输入，然后……
```

推理模型时代的正确策略：
```
# API 参数层
reasoning_effort: "medium"  # 或 effort: "high" 或 thinking_level: "MEDIUM"

# 提示词层
分析此代码库的安全漏洞。在完成之前，验证每个发现是否有具体的代码证据支撑。
```

**构建指引**：

1. **从最低推理级别开始**：GPT-5.4 建议"大多数团队应从 none、low 或 medium 开始"。先测试低推理级别，仅在质量不足时提升。
2. **用"initiative nudge"替代提升推理级别**：GPT-5.4 引入了一个精妙的策略——在提升 reasoning_effort 之前，先尝试添加"不要停在第一个看似合理的答案"这样的鼓励性指令 。
3. **避免显式 CoT 诱导**：在推理模型上，"让我们一步步思考"不仅多余，甚至有害。推理模型已经在内部自动执行了基于强化学习的思维搜索，外部的 CoT 指令会迫使模型浪费资源将内部的高维推理过程翻译成低效的人类自然语言。
4. **自检指令仍然有效**：Anthropic 建议"在完成之前，验证你的答案是否符合 [测试标准]"——这不是要求模型"如何思考"，而是定义"思考的终止条件"。
5. **利用平台特有机制**：
   - GPT-5.4：使用 Responses API 传递 `previous_response_id`，使模型能够引用前一轮次的推理链，减少重复推理。
   - Claude Opus 4.6：信任自适应思考机制的自动校准，仅在需要限制延迟时手动降低 effort。
   - Gemini 3.1 Pro：对需要低延迟的场景，设定 `thinking_level: LOW` 并在系统指令中加入"think silently"。

**校准矩阵**：

| 任务类型 | GPT-5.4 推荐 | Claude 4.6 推荐 | Gemini 3.1 Pro 推荐 |
| --- | --- | --- | --- |
| 执行型（字段提取、格式转换、简单转换） | reasoning_effort: none | effort: low / 自适应 | thinking_level: LOW |
| 分析型（代码审查、Bug 定位、文档分析） | reasoning_effort: low / medium | effort: medium / 自适应 | thinking_level: MEDIUM |
| 研究型（长上下文综合、多文档审查、策略设计） | reasoning_effort: medium / high | effort: high / 自适应 | thinking_level: HIGH |
| 极端推理（复杂数学、多步证明、架构级决策） | reasoning_effort: high / xhigh | effort: max | thinking_level: HIGH |

#### 层 5：验证与迭代层（Verification & Iteration Layer）

**功能**：定义输出的验证标准、自检指令和迭代协议。这一层确保模型的输出不仅"看起来正确"，而且经过了内部或外部的质量检验。

**跨平台映射**：
- **Anthropic**："在完成之前，验证你的答案是否符合 [测试标准]……这能可靠地捕获错误，尤其在编程和数学任务中" 。同时推荐"链式提示的最常见模式是自我纠正：生成草稿 → 根据标准审查 → 根据审查结果改进" 。
- **OpenAI**："建立评估（evals）来衡量提示词的行为……优化模型输出需要评估、提示词工程和微调的飞轮反馈" 。GPT-5.4 指南特别强调"定义什么算作完成，以及模型应如何验证其工作" 。
- **Google**："提示词工程是测试驱动和迭代的过程……重要的是清晰地定义目标和预期结果，并系统性地测试它们" 。同时推荐**分步验证**（Split-step Verification）——"首先验证能力存在，然后再生成答案" 。

**构建指引**：

1. **嵌入自检指令**：在提示词末尾添加验证要求，如"在提交答案前，验证每个声明都有来源支撑"或"检查代码是否处理了所有边缘情况"。
2. **对比少样本验证**（Contrastive Few-Shot）：同时展示一个正确示例和一个典型错误示例，并标注错误原因。研究表明，这种嵌入在代码上下文中的对比性负向约束，比抽象的全局规则更精确有效。
3. **链式自我纠正**：对关键任务采用"生成 → 审查 → 改进"的三步链式结构，每步使用独立的 API 调用以便检查中间输出。
4. **测试驱动后置条件**：要求生成的代码附带验证测试，确保正向输入被接受且负向条件被拒绝。

**底层支撑——MAPS 与推理模型的协同**：

多层自我反思框架（MAPS）提供了一种强大的验证机制：当模型生成了有缺陷的输出时，不使用通用的反思指令，而是强迫模型进入"元状态"，动态编写一条定制化的反思指令。实证评估表明，2-3 层反思深度的 MAPS 框架能将准确率从 76.1% 推至 95.5%。

然而，在推理模型时代，MAPS 的角色需要被重新审视。推理模型的内部推理过程本身已经包含了隐式的自我反思（DeepSeek-R1-Zero 在纯 RL 训练中自主涌现了反思和自我修正能力）。因此，对于推理模型，MAPS 的显式多轮反思可能与内部推理产生冗余。更优的策略是**将 MAPS 的核心思想（定制化反思指令）融入验证层的自检指令中**，让推理模型在单轮推理内完成反思，而非通过多轮 API 调用。

### 3.3 层间交互与排序原则

**物理排序**：UPGA 的五层在提示词中的推荐物理排列顺序为：

```
[层2: 上下文架构] ← 静态参考内容，放在最前面
[层1: 意图声明] ← 核心目标和范围
[层3: 行为规约] ← 格式约束和行为指令
[层4: 推理委托] ← 自检指令和推理引导（API参数在代码层设置）
[层5: 验证与迭代] ← 验证标准，放在最后面
```

注意层 2（上下文）放在层 1（意图）之前——这是基于三大平台共同推荐的"上下文前置、查询后置"原则。层 5（验证）放在最末尾，确保模型在生成最终输出前最后看到的指令是验证标准。

**最小必要层原则**：

UPGA 的五个层是**可选组合的，而非强制全部使用**。对于简单的执行型任务（如"将此 JSON 转换为 CSV"），可能只需要层 1（意图声明）和层 3（行为规约）。强制使用所有五层会导致过度工程化——这正是三大平台共同警告的反模式：
- Anthropic 提供了专门的"overengineering prevention"指令模板 。
- OpenAI 建议"在提升 reasoning_effort 之前先尝试 initiative nudge" 。
- Google 警告推理模型"可能会过度分析冗长的传统提示词技术" 。
