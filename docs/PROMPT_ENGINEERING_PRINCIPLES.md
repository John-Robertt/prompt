# 提示词工程核心原则

> 面向 AI 编程场景的提示词设计框架：5 条原理 + 5 步编写顺序 + 质量自检表。

## 读者与目标

- **读者**：编写 AI 编程系统提示词、CLAUDE.md、Agent 指令的工程师
- **目标**：读完能写出减少模型不确定性、结构清晰、可验证的提示词
- **验收**：用文末自检表对照你的提示词，5 条全绿即为合格

## 底层事实

LLM 是概率引擎：给定输入 token 序列，预测下一个 token 的条件概率分布。提示词的全部作用是**塑造这个分布的形态**——让正确路径更陡、更集中，让错误路径更难被采样。

因此，提示词的核心任务不是"激励模型"，而是**减少不确定性**。模型最怕的是歧义、噪声、冲突和注意力分散；最强提示词不是最花哨的，而是最能让模型少猜的。

## 五条核心原则

### 1. 明确目标与验收标准 —— 定义"做什么"

没有成功定义，模型只能猜你想要什么。一旦写清输出目标、边界条件和完成标准，模型行为会立刻稳定。这是第一原则——目标不清，其他都无从谈起。

**验收**：独立读者能仅凭提示词复述出期望输出的格式、范围和完成条件。

### 2. 提供充分且相关的上下文 —— 定义"基于什么"

模型不是缺"聪明"，常常是缺"当前任务所需的状态"。代码片段、依赖关系、约束条件、已有规范、输入输出样例，比"你是顶级专家"更重要。上下文不全 → 幻觉；上下文有噪声 → 注意力分散。

**验收**：模型完成任务所需的全部信息（代码、依赖、约束、样例）都已显式包含在提示词中。

### 3. 用正向、具体、可执行的要求塑造输出分布 —— 定义"怎么做"

告诉模型"做什么"比"不要做什么"更稳。因为否定词会激活被禁止概念的概率路径，正向指令则直接铺设高概率的正确路径。具体的动作动词和参数比抽象形容词有效——迭代率从 38.5% 降到 12.3%，违背率从 12% 降到 3%。

**验收**：每条指令都用"做什么"表述；否定表达已替换为正向替代。

### 4. 用结构化隔离组织以上所有内容 —— 确保前三条不失真

前三条是语义内容，结构化隔离是承载容器。XML 标签、分隔符、首尾锚定（书挡策略）做的是命名空间隔离——防止指令、上下文、示例、用户输入在注意力层互相污染。在简单短任务里可选；在长上下文、多规则、代码库级任务里，它不是增益项而是前提条件。简单的位置重排就能提升 30% 准确度。

**以上四条构成单轮提示词的完整设计原则。**

**验收**：指令、上下文、示例、用户输入用标签或分隔符物理隔离。

### 5. 建立验证与修复反馈循环（复杂任务必需）

对复杂任务，单轮提示词无论多完美都无法覆盖所有边界情况。MAPS 框架让 Llama-8B 从 76.1% 跃升至 95.5%，不是靠更好的提示词，而是靠"错了能被针对性纠正"。这一条属于工作流层，不是提示词层，但在 AI 编程系统中不可或缺。

**验收**：提示词或配套工作流中定义了成功标准和出错时的修复路径。

## 原则速查表

常用技巧的正向使用指南：

| 技巧              | 使用场景                                                                           | 跳过场景                                                                          |
| ----------------- | ---------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **角色设定**      | 需要调整输出风格和格式时启用                                                 | 需要提升推理或编程正确性时跳过（编程任务中加角色可能降低性能）           |
| **中性语调**      | 所有技术任务默认使用，去掉情绪噪声，防止情感反弹和基调底线效应               | 创意写作或需要特定语气的场景                                                      |
| **Few-shot 示例** | 目标已清楚时，直接定义输出分布形状——模型最自然的对齐方式                     | 目标本身模糊时先解决原则 1                                                        |
| **思维链**        | 传统模型（GPT-4、Claude 3.5 及更早）启用                                    | 推理模型（o1/o3、Claude 3.7+）跳过，模型已内置且显式 CoT 增加 20-80% 延迟 |
| **上下文工程**    | 规模化系统——固定规则、文档、工具说明的持久化管理                             | 一次性简单任务                                                                    |
| **禁止行为**      | 安全场景必需，但文本层可靠性弱——用"正向替代 + 结构隔离 + 后验检查"组合实现   | 非安全场景用原则 3 的正向表达替代                                                 |

## 编写顺序

原理顺序解释"为什么重要"，编写顺序指导"怎么写不返工"：

1. **目标**：要产出什么
2. **上下文**：模型缺什么信息
3. **搭结构骨架**：用标签/分隔符定好容器
4. **填正向要求与约束**：具体、可执行的指令
5. **验收/验证**：怎样算成功，错了怎么修

## 质量自检表

用以下清单逐条检查你的提示词：

- [ ] **目标**：独立读者能仅凭提示词复述出期望输出的格式、范围和完成条件
- [ ] **上下文**：模型完成任务所需的所有信息（代码片段、依赖、约束、样例）都已包含
- [ ] **正向表达**：每条指令都用"做什么"而非"不要做什么"表述；否定表达已替换为正向替代
- [ ] **结构隔离**：指令、上下文、示例、用户输入用标签或分隔符物理隔离
- [ ] **验证**：定义了成功标准和出错时的修复路径

任一项不通过 → 回到对应原则修正。
