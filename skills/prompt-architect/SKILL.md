# Prompt Architect

## 这个 skill 要解决什么问题

开发者编写的提示词普遍依赖规则枚举和试错调参。这导致提示词在未预见的场景中脆弱失效，在模型版本迭代后漂移退化，在复杂任务中因规则冲突而崩溃。根本原因是：提示词中的指令缺少可追溯的理由，模型只能字面执行而无法理解意图。

**成功标准**：本 skill 生成的提示词满足三个可观察条件——(1) 模型能在指令未显式覆盖的场景中从动机自主推导正确行为；(2) 提示词的每个组件都能追溯到它要解决的现实问题；(3) 提示词能通过实践结果检验并在迭代中持续改进。

## 核心哲学

本文件的所有行为规范溯源至两个基础命题：

> **目的**：提示词是用思想改造现实的工具。它从现实问题中来，回到现实结果中接受检验。

> **机制**：每条指令的力量不来自它自身的措辞，而来自它背后可追溯的理由。理由使指令获得在未知场景中被重新推导的能力。

两个命题的关系：目的命题是根基，机制命题是手段。目的命题决定提示词为什么存在（解决现实问题），机制命题决定提示词如何有效（通过可追溯的理由）。可追溯的理由之所以重要，是因为它使提示词在面对现实问题时更有效——而非因为可追溯性本身是美德。

一条孤立的规则只能覆盖它字面描述的场景；一条携带理由的指令能被模型在任何相关场景中重新推导。理由的组织形态可以是树状演绎（从一个锚点向下展开）、有向图（多条理由交叉支撑）、或其他适合目标场景的拓扑。形态服从内容，不存在唯一正确的结构。

机制命题基于三个认知事实：

- 规则系统在边缘情况呈指数增长时必然崩溃（上下文腐败），而携带理由的指令对环境漂移具有天然免疫力——模型依据理由而非关键字做判断
- 当模型理解了约束背后的理由，会自发生成提示词作者未显式要求的防御性行为（实证：理解"物理隔离实验室"这一理由后，模型自发生成错误捕获、日志脱敏、内存泄漏防护）
- 模型的注意力预算有限，理由充当注意力过滤器，将计算资源集中于验证行为是否对齐于源头意图，而非盲目预测统计上最常见的下一个 Token

## 设计原则

_以下五条原则从核心哲学推导而来，是所有生成行为的根。原则 A 从目的命题推导，原则 B–E 从机制命题推导。遇到原则未覆盖的灰色地带时，回到核心哲学做判断。_

### A. 从问题出发，用结果检验

_← 目的命题：提示词从现实问题中来，回到现实结果中接受检验_

因为提示词是关于"如何解决这个问题"的假说而非终态制品，它的正确性由现实结果裁定——不是内部的逻辑自洽，不是结构的完整优美，不是技巧的精妙堆砌。不存在脱离具体问题的"好提示词"：同一技巧在一个问题面前有效，换一个问题可能完全失效。当结果不符预期时，首先修正思想和动机层（是否对问题的理解有偏差？），而非在行为层堆叠更多规则。

### B. 意图先于指令

_← 机制命题：指令的力量来自理由，而非措辞_

因为解释"为什么"能将模型从统计模式匹配（系统 1）切换至分析性推理（系统 2），每条行为规范都必须携带可追溯的动机链。没有动机支撑的规则是游离的教条——它会在上下文压力下最先被遗忘，在边缘情况下最先失效，在模型版本迭代后最先漂移。

### C. 正向路径是唯一可靠路径

_← 机制命题：可追溯的理由顺着正向路径构建，而非逆向排除_

因为 LLM 的自回归解码天然是正向选择引擎——否定指令（"不要做 X"）会激活被禁止概念的神经表征，迫使模型在每个解码步计算额外惩罚项，长上下文中这一机制极易失效（实证：将"不要做 X"转化为"做 Y"使违背率从 12% 降至 3%）。定义目标状态永远比列举禁止状态更可靠。

### D. 结构是注意力的导航系统

_← 机制命题：理由链需要物理结构来维持其可追溯性_

因为模型的注意力存在 U 型衰减曲线（记住首尾、遗忘中间），且长上下文是信息坍塌的温床。物理结构（标签层级、位置排列、动静分离）直接控制信息的可达性（实证：静态背景知识置前、动态任务指令置后，代码生成准确度最高提升 30%）。

### E. 克制是质量的前提

_← 机制命题：理由的力量来自精准，而非数量_

因为每增加一层提示技巧都在稀释注意力预算：角色扮演不增强逻辑能力（实证：162 种角色无稳定提升，13.78% 的正确答案因引入角色而退化）；情感语言触发"情感反弹"（模型过度安抚而淡化真实缺陷）和"基调底线"（回避不悦但正确的信息）；冗余约束加速上下文腐败。用中性动作动词（"重构"、"提取"、"验证"）驱动指令，用参数化约束（"≤3 层嵌套"）替代抽象形容词（"保持简洁"）。

## 工作流程

收到提示词设计或优化请求后，按以下阶段执行。

如果是**生成新提示词**：从阶段一开始，逐阶段推进。
如果是**优化现有提示词**：先通读全文，识别结构中缺失或违背的设计原则，再从对应阶段介入重构。

### 阶段一：问题锚定 ← 原则 A

在生成任何内容前，首先明确这个提示词要解决的现实问题：

1. **现实问题**：这个提示词要解决什么具体问题？问题的根源是什么？
2. **成功标准**：问题被解决的标志是什么——不是提示词结构上的标志，而是现实中可观察的结果
3. **目标环境**：提示词被哪个模型/平台消费？（系统提示词、开发者消息、CLAUDE.md、Agent 指令等）运行模式是单轮对话、多轮会话、还是自治 Agent 循环？
4. **硬性边界**：不可违背的约束（安全、合规、格式、性能预算、token 上限）
5. **受众与上下文**：提示词的输出面向谁？目标模型的特性是什么？（推理模型需要不同的策略）

如果信息不足以做出架构决策，向用户追问。不在模糊的地基上建造。

### 阶段二：思想锚点识别 ← 原则 A、B

识别能回应现实问题的最底层思想——后续所有指令的根基。

锚点必须回答"为什么"：为什么这个问题需要这样解决？锚点可以是一个（最常见，如"决策质量取决于决策之前的布局"），也可以是少数几个彼此独立的基础理由（如安全性、可用性、合规性各自成为独立锚点）。数量不是目标，可追溯性才是：每个锚点都必须是不可再从其他锚点推导出的基础理由。

检验标准：

- 每个锚点能用一句话表达，回答"Why"而非"What"或"How"
- 锚点与阶段一的现实问题之间存在直接的因果关联——它是对问题的思想回应
- 当遇到规则未覆盖的灰色地带时，锚点（单独或组合）能提供判断依据
- 锚点之间无推导关系（若 A 可从 B 推出，则 A 不是锚点）

辅助提炼工具：第一性原理思维（剥离类比和行业惯例，回到最基础事实）、逆向思维（"如何确保这个系统彻底失败？"然后反转）、二阶思维（推演长期次生后果）。

### 阶段三：理由网络构建 ← 原则 B、C

从锚点出发，构建中间层理由（动机/原则），连接锚点与最终行为。每条中间理由满足：

- **可追溯**：能溯源至一个或多个锚点（"因为[锚点 X]，所以[此理由成立]"；也允许"因为[锚点 X]+[锚点 Y]的交叉要求，所以..."）
- **正交性**：覆盖不同维度，无语义重叠
- **可验证**：能通过具体场景判断某行为是否符合此理由
- **正向构建**：将所有"不要做 X"转化为"做 Y，因为[理由]"

常见的有效模式是树状演绎：一个锚点 → 少量正交原则 → 具体行为。但当目标场景存在多个独立的基础关切时（如安全 + 效率 + 合规），理由网络自然会形成有向无环图——行为可以同时溯源至多条理由。形态服从内容，选择最自然的拓扑。

### 阶段四：行为生成 ← 原则 C、E

从理由网络推导具体行为规范。每条规范满足：

- **正向指令**：定义目标状态，而非列举禁止状态
- **理由溯源**：携带"因为[理由]"标注或"← 理由 X"回溯；允许溯源至多条理由（如"← 原则 A + 原则 B"）。最终输出中保留程度视简洁性需要决定，但设计阶段必须完整
- **参数化**：用具体约束替代抽象形容词
- **中性语调**：动作动词驱动，无情感修饰、无角色能力幻想、无冗余礼貌用语

当负向边界不可避免时（安全、合规等绝对红线）：

- 物理隔离于专用区块
- 置于提示词首尾的注意力高峰区（书挡化策略）
- 配合对比示例（正确做法 + 错误做法 + 错误原因标注）而非仅依赖抽象禁令

**对目标模型的意图传递**：生成的提示词不仅要让目标模型知道"做什么"，还要让它理解"为什么这样做"。因为目标模型理解了理由，才能在指令未覆盖的场景中自主推导正确行为，在指令冲突时依据上游动机裁决优先级。

### 阶段五：结构封装 ← 原则 D

将内容组装为结构化提示词：

1. **格式选择**：根据目标环境选择 Markdown 层级、XML 标签或混合格式
2. **信息拓扑**：
   - 锚点和全局理由 → 序列开头（建立认知锚点）
   - 大段静态背景知识 → 紧随其后（可利用提示词缓存机制）
   - 具体行为规范按主题分组 → 中段（每组标注原则溯源）
   - 动态任务指令 → 序列末端
   - 关键不可违约束 → 首尾双重锚定
3. **推理模型适配**：面向原生推理模型（o1/o3/Claude 3.7+）时，移除显式思维链诱导（"一步步思考"），改为定义清晰的成功验收标准，让模型自主展开内部推理
4. **冗余压缩**：删除不携带独立信息的填充语句；合并语义重叠的规范

### 阶段六：验证 ← 原则 A、E

对生成的提示词执行双重检查——先检查内部结构，再检查对外部问题的回应性。

**内部结构检查：**

- [ ] **理由可追溯性**：每条行为规范至少溯源至一条中间理由，每条中间理由至少溯源至一个锚点；不存在无法追溯的游离规则
- [ ] **正向率**：负向指令占比 < 20%，且每条负向指令有不可替代的存在理由
- [ ] **结构清晰度**：信息边界通过标签或层级明确划分，无混淆区域
- [ ] **中性语调**：无情感修饰词、无角色能力幻想、无冗余礼貌用语
- [ ] **参数化**：无抽象形容词充当约束（"保持简洁"→ 具体数值或结构要求）
- [ ] **注意力经济**：无冗余重复、无不携带独立信息的填充句、无过度堆砌的提示技巧组合

**外部问题回应性检查：**

- [ ] **问题锚定**：提示词中的每个组件都能追溯到阶段一定义的现实问题；不存在与问题无关的"最佳实践"堆砌
- [ ] **成功标准对齐**：提示词的行为规范在被正确执行时，能导向阶段一定义的可观察成功标准
- [ ] **抗漂移性**：思想锚点和动机层表述不依赖特定关键字匹配，环境变化时仍能指导行为推导
- [ ] **意图可传递**：目标模型能从提示词中理解"为什么这样做"，而非仅获得"做什么"的指令

发现问题时，定位到对应的设计原则，从动机层面修正——而非在行为层面打补丁。

### 阶段七：实践检验 ← 原则 A

提示词交付后进入实践检验循环。提示词是假说，不是终稿。

#### 7.1 构建测试场景

从阶段一的成功标准出发，设计 2–3 个能检验提示词有效性的真实场景：

- **覆盖核心路径**：至少一个场景直接对应提示词要解决的主要问题
- **覆盖边缘情况**：至少一个场景涉及指令未显式覆盖的灰色地带——检验动机推理链是否赋予了模型"举一反三"的能力
- **建立基线对比**：在相同场景下，对比"使用此提示词"与"不使用此提示词"的模型输出差异，验证提示词是否带来了增量价值而非仅仅增加了上下文负担

向用户确认测试场景是否反映了他们面对的真实问题。测试场景如果脱离现实，验证就失去意义。

#### 7.2 运行与观察

将提示词投入测试场景运行，观察两个维度：

- **结果维度**：模型输出是否达成阶段一定义的可观察成功标准？
- **过程维度**：模型的执行路径是否体现了对动机的理解？还是仅在字面上遵循了行为规则却偏离了意图？（过程正确但结果错误说明思想层可能有偏差；结果正确但过程机械说明动机传递可能不足）

#### 7.3 偏差诊断

当结果不符预期时，按以下路径**自顶向下**逐层排查，在定位到的层级修正：

1. **思想层**：对问题的理解是否有偏差？锚点是否回应了真正的问题？——如果是，回到阶段一和阶段二
2. **动机层**：理由网络的推导链是否断裂？动机之间是否存在未察觉的冲突？——如果是，回到阶段三
3. **行为层**：具体措辞或结构是否导致模型误解？——如果是，修正阶段四和阶段五的产出

核心纪律：**在行为层打补丁是最后手段，不是第一反应。** 如果发现自己在反复添加规则来修补输出问题，这通常说明问题出在上游的思想或动机层。

#### 7.4 迭代

修正后重新进入 7.2 运行。每轮迭代保留记录（修正了什么、为什么、结果如何变化），使改进过程可追溯。

#### 7.5 经验沉淀

目标达成后，总结哪些思想和动机在实践中被验证——这些经验可迁移到同类问题的未来提示词中，但不可不加检验地照搬到不同性质的问题上。同时标注哪些最初的假设被实践推翻了，这些失败经验同样有迁移价值。
