## 核心哲学：程序 = 数据结构 + 算法

### 第一原理：数据结构决定一切

- **数据即程序架构** - 正确的数据结构让代码变得不言自明。
- **关系即逻辑** - 数据之间的关系就是程序的核心逻辑。
- **结构即性能** - 数据的组织方式直接决定算法效率。

### 第二原理：算法服务于数据结构

- 算法不是为了炫技，而是为了最高效地操作已定义的数据。
- 最好的算法是那些让数据结构的天然特性发挥到极致的算法。

## 工作流程

请严格按照以下步骤创建 TODO List 分析任务：

### 步骤 1：业务需求验证

回答以下根本问题：

- 实际业务的当前状态是什么？（现状）
- 用户真正想要解决的核心问题是什么？（痛点）
- 这是真实存在且高频的问题，还是一个臆想出来的边缘情况？（价值）

### 步骤 2：数据结构探索

回答以下问题：

- 该问题的核心数据是什么？（实体识别）
- 这些数据之间的关系是什么？（关系建模）
- 数据的生命周期是什么？（状态管理）

**请使用此模板进行分析：**

```
核心数据实体 (Core Data Entities)：
- 实体A：[描述]
- 实体B：[描述]

数据关系图 (Data Relationship Diagram)：
- A → B：[关系类型，如 '包含', '引用', '拥有']
- B → C：[关系类型]

- 关系基数 (Cardinality):
- A 与 B 的关系是：[一对一 (1:1), 一对多 (1:N), 多对多 (M:N)]
- ...

数据生命周期 (Data Lifecycle)：
创建 → 读取 → 更新 → 删除 (CRUD)
[每个阶段谁负责？如何保证数据一致性 (Consistency)？]
```

### 步骤 3：数据结构设计

根据你的数据分析，设计最直接的数据表示。遵循以下原则：

- **扁平化 (Flattening)**：避免不必要的嵌套和间接引用。
- **局部性 (Locality)**：将相关数据在物理内存上组织在一起。
- **最小化 (Minimization)**：只保留真正需要的字段。
- **不变性 (Immutability)**：尽可能使数据结构不可变，以简化并发和状态管理。

### 步骤 4：算法选择（诊断式漏斗）

**请回答以下诊断问题：**

1.  **操作频率分析 (Operation Frequency Analysis):**
    - **最高频的操作是什么？** 是读取（单点查找 vs 范围扫描）、写入（追加 vs 随机插入）、还是更新或删除？
    - **读写比例是多少？** (e.g., 90% 读, 10% 写)
2.  **数据特征分析 (Data Characteristics Analysis):**
    - 数据集合的大小是固定的还是动态变化的？
    - 数据本身是否有序？或者是否需要维持其顺序？
3.  **访问模式分析 (Access Pattern Analysis):**
    - 是否需要通过索引进行快速随机访问 (Random Access)？（例如 `array[i]`）
    - 还是只需要顺序遍历 (Sequential Traversal)？（例如 `for item in list`）
4.  **约束条件分析 (Constraints Analysis):**
    - **内存限制 (Memory Constraints):** 空间复杂度是否是首要考虑因素？
    - **延迟要求 (Latency Requirements):** 操作必须在特定时间内完成吗？

**基于以上诊断，对你的算法选择进行评估：**

1.  **时间复杂度 (Time Complexity)** - 是否与最高频操作的需求匹配？
2.  **空间复杂度 (Space Complexity)** - 是否在内存限制之内？
3.  **代码复杂度 (Code Complexity)** - 实现是否直观？是否引入了不必要的复杂性？

### 步骤 5：实现验证

使用此清单进行验证：

- ✅ 数据结构之间有清晰的、最小化的关系。
- ✅ 核心算法完全由最高频的数据操作需求驱动。
- ✅ 算法充分利用了所选数据结构的内在优势（例如，利用哈希表的 O(1) 查找）。
- ✅ 没有不必要的数据转换或“胶水代码”。
- ✅ 没有隐藏的性能陷阱（如在循环中进行数据库查询、不经意的全表扫描等）。
- ✅ 代码的结构直接反映了数据的关系。
